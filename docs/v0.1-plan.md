# v0.1 Plan

## Problem

Kevin wants AI to think proactively, find useful work, and avoid waiting for
explicit commands, while still preserving safety and Kevin's decision style.

## User Goal

An always-available assistant that notices useful work, ranks it, and prepares
safe next actions using Kevin's persona.

## MVP Scope

1. Run as a Docker service so the same assistant can be deployed on different
   hosts.
2. Load Kevin persona from a configured rule-source location.
3. Read `homelab-docs` and optional user-selected rule sources without hard-coded
   machine paths.
4. Apply Kevin's persona consistently while allowing environment-specific rules
   to shape what can be observed or proposed.
5. Observe all configured services in read-only mode before suggesting work.
6. Scan repositories in read-only mode.
7. Produce candidate tasks.
8. Classify tasks by autonomy and risk.
9. Generate OpenCode prompts.
10. Produce a concise Traditional Chinese report.

## Deployment Assumption

v0.1 is designed as a Dockerized observer. Configuration must provide paths or
mount names for rule sources, repositories, and service definitions. The service
must not assume `D:\Projects\_HomeProject`, `D:\GitClone\_HomeProject`, or any
single host layout exists.

The initial deployment target is `kevinhome` because it already hosts the local
HomeProject repositories and OpenCode. This is a starting environment choice,
not an application-level path assumption.

Recommended container inputs:

1. Read-only mount for `homelab-docs`.
2. Optional read-only mounts for additional rule repositories selected by Kevin.
3. Read-only mounts or API access for repositories to scan.
4. A config file listing environments, services, repositories, and allowed health
   checks.
5. A writable data volume only for Autopilot's own SQLite task/report store.

## First Phase: Service Observation

The first implementation phase should observe the whole configured service set
before proposing fixes. Observation means collecting safe, read-only facts:

1. Service name, domain, host, port, and source document.
2. Repository location and current branch when a repo is configured.
3. Git dirty status and recent commit summaries.
4. Declared Docker or compose metadata from non-secret files.
5. Health/status endpoint result when explicitly allowed.
6. Known limitations and stale-doc signals from rule/docs files.

Observation must not read `.env*`, credential JSON, service-account files, secret
paths, or container environment variables.

## Candidate Task Sources

1. Service observation gaps or unhealthy allowed endpoints.
2. TODO/FIXME markers.
3. Dirty git worktrees.
4. Recent failed checks if logs are available.
5. Docs that mention old behavior or stale paths.
6. Missing README/setup instructions.
7. Repeated manual workflows described in docs or issues.

## Task Scoring

```text
score =
  user_experience_value
+ stability_value
+ verifiability_value
+ automation_value
+ prototype_value
- regression_risk
- approval_required_weight
- cost_or_deployment_risk
```

## v0.2: Idea Intake

v0.2 starts the idea handoff surface. Kevin can paste a rough idea into the Web
dashboard; Autopilot stores the raw idea, classifies it, records whether the
classification came from ai-core or deterministic fallback, and suggests safe
next steps. It still does not create repositories, write product code, deploy,
commit, or push.

AI thinking uses `@kevinsisi/ai-core` pinned to commit
`f42e3f4ceb14886604bd0c7f248071018c85ff82`. If `GEMINI_API_KEY` or
`GOOGLE_API_KEY` is unavailable, v0.2 preserves the idea and uses deterministic
fallback classification.

## Future: Full Idea Handoff Mode

Kevin often has many new project ideas during the day. Autopilot should become a
handoff surface where Kevin can paste a rough idea and let the system continue
the workflow.

Target flow:

1. Preserve Kevin's raw idea and infer the user pain or workflow goal.
2. Ask only missing product/safety questions that cannot be researched.
3. Recommend whether to use an existing repo or create a new repo.
4. Recommend deployment target based on performance, uptime, data locality, and
   safety.
5. Draft architecture and stack using HomeProject standards.
6. Create or update OpenSpec before implementation.
7. Generate bounded OpenCode prompts for implementation.
8. Run tests/builds and report evidence.
9. Prepare commit, push, deployment, and post-release verification steps when
   the approved autonomy level allows them.

The full handoff workflow is not part of v0.2 execution. It should be designed as
a later approved mode because it can create repos, change infrastructure, deploy,
and affect user flows.
